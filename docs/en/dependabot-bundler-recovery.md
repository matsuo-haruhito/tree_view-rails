# Dependabot Bundler Recovery

Use this note when a Dependabot Bundler pull request changes `Gemfile.lock` and fails before the regular lint baseline is the useful signal.

## Recognize lockfile metadata drift

The CI policy command includes `script/test_gemfile_lock_dependency_drift.mjs`. That smoke compares direct `Gemfile` requirements with the `Gemfile.lock` `DEPENDENCIES` metadata and fails with this message when the lockfile metadata is stale:

```text
Gemfile.lock DEPENDENCIES must match direct Gemfile gem requirements; run bundle install after changing Gemfile dependency metadata
```

Bundler frozen install can report the same class of failure before Ruby lint reaches the repository source. Treat this as lockfile metadata drift, not as the Standard / RuboCop baseline drift tracked separately in #2150.

## Choose the recovery path

- Use `@dependabot recreate` when the Dependabot branch only contains the dependency update and the lockfile metadata should be regenerated by Dependabot from the current base branch.
- Use a normal rebase only when the Dependabot branch has not been edited by a human or agent and Dependabot can still safely refresh it.
- Use a manual lockfile refresh when a maintainer intentionally owns the branch or needs to keep additional reviewed changes. Run `bundle install`, commit the resulting `Gemfile.lock`, and re-run the CI policy smoke before relying on `bundle install` in frozen mode.

## Review points

Confirm the failing job and the exact mismatch first. If the pull request fails only because `Gemfile.lock` metadata is stale, keep the recovery scoped to the lockfile and do not mix it with broad lint cleanup, dependency grouping changes, or CI workflow redesign.

After recovery, re-check the pull request head SHA. If `bundle exec standardrb` then reports `Style/RedundantStructKeywordInit`, continue to treat that as the separate #2150 lint baseline issue rather than a Bundler recovery problem.
