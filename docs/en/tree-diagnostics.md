# Tree diagnostics

This page explains diagnostics helpers for inspecting TreeView structure and investigating data issues.

## Overview

Tree diagnostics help inspect tree structure, data consistency, and rendering risk.

They are useful for checking:

- node key uniqueness
- DOM ID collisions
- orphan records
- parent paths
- tree stats
- cycle diagnostics
- expanded keys for items

Use them during development, before release, or when investigating inconsistent host-app data.

## Recommended validation entry point

When integrating TreeView into a host app, start with this pre-render validation flow:

1. Build the tree with `validate_node_keys: true` so duplicate node keys fail early.
2. Decide an `orphan_strategy:` that matches the host app's data policy.
3. Inspect `tree.orphan_items` when records can be filtered, imported, or permission-scoped.
4. Build the `RenderState` with the same `tree`, `root_items`, `row_partial`, and `ui_config` the page will render.
5. Call `render_state.validate_unique_dom_ids!` in development, tests, or pre-release checks.
6. Use `TreeView::CycleDiagnostics.new(tree).report` when parent relationships come from user-editable or imported data.

A minimal pre-render check can look like this:

```ruby
tree = TreeView::Tree.new(
  records: documents,
  parent_id_method: :parent_document_id,
  validate_node_keys: true,
  orphan_strategy: :raise
)

render_state = TreeView::RenderState.new(
  tree: tree,
  root_items: tree.root_items,
  row_partial: "documents/tree_columns",
  ui_config: tree_ui
)

render_state.validate_unique_dom_ids!
```

This gives host apps a concrete validation point before rendering invalid or surprising tree data.

## Node key uniqueness

Enable validation when building the tree to catch duplicate node keys early.

```ruby
tree = TreeView::Tree.new(
  records: documents,
  parent_id_method: :parent_document_id,
  validate_node_keys: true
)
```

Duplicate keys raise `ArgumentError`. The usual fix is to provide a stable, collision-resistant key strategy, such as `id_method:` in records mode or `node_key_resolver:` in resolver / adapter mode.

## DOM ID collisions

Use `RenderState#validate_unique_dom_ids!` to check generated DOM IDs.

```ruby
render_state.validate_unique_dom_ids!
```

The validation includes DOM IDs generated by configured features such as selection checkboxes. If this fails, check `node_prefix`, custom DOM ID builders, and node keys that may normalize to the same browser-facing ID.

## Orphan diagnostics

In records mode, use orphan helpers to find records whose parent is missing from the record set.

```ruby
tree.orphan_items
```

`orphan_strategy` controls how orphans are handled: ignored, shown as roots, shown as orphans only, or raised as errors.

```ruby
tree = TreeView::Tree.new(
  records: documents,
  parent_id_method: :parent_document_id,
  orphan_strategy: :as_root
)
```

Choose `:raise` in tests or import validation when orphan records should be treated as invalid data. Choose `:as_root` only when showing permission-scoped or filtered subsets as independent roots is intentional.

## Parent path helpers

Records mode can inspect parent paths.

```ruby
tree.parent_for(document)
tree.ancestors_for(document)
tree.path_for(document)
tree.paths_for(documents)
```

Use path helpers to build expanded keys for search results or current nodes.

```ruby
expanded_keys = tree.expanded_keys_for([document])
```

## Tree stats

Use stats helpers to inspect tree size and reachable nodes.

```ruby
tree.stats
```

These signals can help decide whether to use render scope, windowed rendering, or lazy loading for large trees.

## Cycle diagnostics

Cycles in parent relationships can break descendant counts or path traversal.

Cycle diagnostics help investigate invalid parent-child data during development.

```ruby
TreeView::CycleDiagnostics.new(tree).report
```

If a report identifies a cycle, fix the host app data or resolver logic before rendering. TreeView can report the risk, but the host app owns the data correction policy.

## Pre-production checklist

Before enabling a new tree in production, verify:

- Node keys are stable across requests and unique for all rendered nodes.
- Heterogeneous nodes include a class name, namespace, or equivalent discriminator in their key.
- Orphans are either expected and documented, or rejected with `orphan_strategy: :raise`.
- DOM IDs validate with the same `UiConfig` used by the page.
- Initial expansion and persisted state use tree node keys, not UI-only DOM IDs.
- Large tree stats have been reviewed against the chosen render strategy.
- Imported or user-editable parent relationships have been checked for cycles.

## When to use diagnostics

- Check DOM ID and node key collisions before release.
- Inspect orphans after imports or migrations.
- Build expanded keys for initial expansion to search results.
- Decide rendering strategy for large trees.
- Investigate invalid parent relationships in host-app data.

## Responsibility boundary

| Area | TreeView | Host app |
|---|---|---|
| diagnostics helper | yes | calls helper |
| duplicate key detection | yes | chooses stable keys |
| DOM ID collision detection | yes | fixes config/prefixes |
| orphan detection | yes | decides data policy |
| cycle report | yes | fixes data |
| rendering strategy decision | provides signals | chooses behavior |
