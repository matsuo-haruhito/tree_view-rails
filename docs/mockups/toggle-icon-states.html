State-based icon comparison

The toggle cell keeps the same hierarchy branch slots while the visible icon changes by state. Copy and final icon artwork remain host-app responsibilities.

tree icon state generated boundary review note
- Workspace root expanded
  • toggle_icons[:expanded]
  • aria-expanded="true"
 Expanded rows still expose the collapse action; icon shape should not be the only cue.
+ Planning folder collapsed
  • toggle_icons[:collapsed]
  • aria-expanded="false"
 Collapsed rows keep the branch path visible before the icon so depth stays readable.
leaf Notes record leaf
  • toggle_icons[:leaf]
  • no expand action
 Leaf rows can carry a placeholder icon, but should not look actionable unless the host app adds an action.
... Loading children loading
  • toggle_icons[:loading]
  • aria-busy="true"
 Loading icon is a temporary state cue; fetch timing and retry behavior stay outside this mockup.

Depth and type variation

This pass shows how state icons can vary by depth or node type while using the same table and branch structure. It is a visual boundary reference, not a shipped icon pack.

tree variation source review note
R-Root type by_type[:root][:expanded] Type-specific root icon can be distinct without changing row structure.
F+Folder type by_type[:folder][:collapsed] Folder icon can communicate grouping while final labels and color stay host-app owned.
RRecord type by_depth[2][:leaf] Depth variation should help scanning, not replace the branch line or accessible label.

Responsibility boundary

TreeView-owned

Stable slots, state lookup order, branch structure, and rendered action or leaf placement.

Host-app-owned

Final icon set, color tokens, business copy, tooltip wording, and product-specific state meaning.

Accessibility

Do not rely on icon shape alone. Pair icon changes with text, ARIA state, or host-owned labels where needed.
