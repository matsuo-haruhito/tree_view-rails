Diagnostics and integration troubleshooting

Static reference for reviewing two setup-time surfaces: pre-render diagnostics signals and CSS / JavaScript wiring symptoms. The examples keep correction policy, asset loading, controller registration, routes, and release gates host-app owned.

Diagnostics signals

Duplicate keys, DOM ID collisions, orphans, and cycle risk are shown as TreeView-provided signals before render.

Integration symptoms

Normal wiring, CSS missing, JavaScript missing, and Turbo route gaps are separated so reviewers do not conflate setup failures.

Host-app boundary

Repair workflow, release gates, asset pipeline choices, Stimulus boot order, and Turbo endpoints stay outside the gem.

TreeView signal

Diagnostics expose reviewable signals. They do not decide whether a release gate blocks, how data is repaired, or which dashboard copy appears.

Same render state

DOM ID checks should use the same tree, root items, row partial, and UI config as the page being reviewed.

Visual scope

The table is a static inspection surface, not a production diagnostics console or validation workflow.

Review checklist

  • Duplicate key, DOM ID collision, orphan, and cycle risk are visually separate and map back to the diagnostics docs.
  • CSS missing and JavaScript missing states are distinguishable without implying TreeView owns host-app asset or Stimulus setup.
  • Turbo route gaps are shown as host-app endpoint / response issues, not as stylesheet or JavaScript registration problems.
  • Runtime Ruby, JavaScript, diagnostics behavior, public API manifest, asset setup, and Turbo endpoints are intentionally unchanged.
