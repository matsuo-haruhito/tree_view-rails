Filtered tree modes mock

Static reference for comparing how filtered trees keep only matches, matches with ancestors, matches with descendants, or matches with both ancestor and descendant context. The page stays focused on reusable TreeView output and keeps search forms, ranking, highlighting, and authorization copy outside the mock.

Mode A

Matched only

Use this when the host app wants the narrowest result set and does not need surrounding hierarchy for review.

Only the matched node stays visible

query: billing mode: matched_only

TreeView keeps the matching row itself. Any explanation of why it matched or how to refine the query stays in the host app.

tree name owner state
leaf match Billing reports Finance matched

Mode B

Matched with ancestors

Use this when reviewers need the path back to the root so the matching result keeps its hierarchy context.

Matched node plus the path that leads to it

query: billing mode: with_ancestors

Ancestors remain visible so the host app can show where the result lives, but the mock still avoids highlighting or ranked result copy.

tree name owner state
open root ancestor Operations Platform context
child match Billing reports Finance matched

Mode C

Matched with descendants

Use this when the host app wants the matching parent to stay visible together with its visible children or follow-on rows.

Matched parent plus the rows underneath it

query: billing mode: with_descendants

Descendants stay visible after the match so reviewers can inspect downstream rows, while sort order and query semantics remain host-app choices.

tree name owner state
open root match Billing Finance matched

child descendant Refund queue Finance visible

Mode D

Matched with ancestors and descendants

Use this when reviewers need both the path to the match and the rows below that matched branch in one filtered output.

Matched branch with both surrounding contexts

query: billing mode: with_ancestors_and_descendants

Ancestor rows show where the match lives, the matched row stays explicit, and descendant rows remain inspectable. Search copy, ranking, highlighting, and permission-aware filtering still belong to the host app.

ReverseTree reference

Reverse path from matched leaf to ancestors

Use this focused section when the review question is child-to-parent path readability rather than search filtering, breadcrumbs, or route navigation.

Matched leaf becomes the visible root of the review path

The matched item is first, then ancestor rows continue underneath it. TreeView owns the row cues; host apps own breadcrumbs, route wording, and final navigation policy.

GraphAdapter reference

Shared and duplicate path boundary

Two paths can point to one logical node without choosing a dedup policy here

What this mock compares

Mode Best for Keep outside scope
:matched_only The tightest result set when only the matched row itself should remain visible. Result ranking, highlighting, and final search-result messaging.
:with_ancestors Keeping the path back to the root visible for hierarchy review. Authorization, breadcrumb wording, and query explanations.
:with_descendants Keeping a matched parent together with visible rows beneath it. Downstream sorting rules, pagination, or fetch behavior.
:with_ancestors_and_descendants Keeping both the path to the match and the rows below the matched branch visible. Search UI, ranked explanations, text highlighting, permission copy, or authorization-aware filtering.
ReverseTree Reviewing child-to-parent path cues with the matched leaf as the first visible row. Breadcrumb routes, navigation IA, or ReverseTree API changes.
GraphAdapter Reviewing shared logical nodes across duplicate graph paths and node-key namespacing cues. Traversal policy, dedup behavior, resolver implementation, authorization, or database query choices.
  • :with_ancestors_and_descendants combines the ancestor and descendant retention cues in one filtered output.
  • ReverseTree and GraphAdapter samples are focused visual references on this existing page, so no new mockup inventory entry is needed.
  • This focused page keeps product behavior outside the mock so reviewers can inspect mode differences without extra search workflow decisions.
