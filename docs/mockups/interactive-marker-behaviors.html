What this mock compares

Surface Best for Keep outside scope
data-tree-view-interactive Showing the broad custom-widget marker when a non-native control should opt out of every TreeView behavior guard. Widget implementation details, final business logic, or route wiring.
data-tree-view-ignore-keyboard Comparing widgets that need arrow, space, or enter for themselves without claiming the broader row-click or drag boundaries. Keyboard policy redesign, shortcuts outside the widget, or product-specific editor behavior.
data-tree-view-ignore-row-click Highlighting the reserved boundary for host-app row-click integrations so one inner control can stay clickable. Implementing the host-app row-click action itself or changing current TreeView runtime policy.
data-tree-view-ignore-drag Showing the narrow drag-start opt-out for sliders, scrubbers, or other controls that only need transfer protection. Drop target rules, persistence, or drag-and-drop workflow design.
Native controls Confirming that standard inputs, buttons, links, selects, textareas, and editable labels already behave like interactive targets. Validation, save timing, or business copy owned by the host app.
  • These markers only opt a control out of TreeView behavior. Validation, persistence, permissions, and CRUD flows remain host-app responsibilities.
  • `data-tree-view-ignore-row-click` is shown as the documented boundary for app-owned row-click integrations, not as a full runtime feature demonstration.
