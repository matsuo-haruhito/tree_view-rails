Post-drop outcome states

These states start after TreeView has already reported a drop. They show reviewable host-app outcomes without turning TreeView into the persistence, authorization, or final copy owner.

  • Use pending and accepted cues to review feedback timing after a host app begins saving the move.
  • Use rejected and retry-needed cues to keep authorization, validation, and recovery copy in the host app.
  • Keep undo, audit logs, reorder policy, and final business wording outside the static TreeView mockup.

Pending save

The host app has accepted the drop event and is saving the requested move.

treeoutcome cue
saveQueued move saving move
  • host persistence
  • TreeView event already sent

Accepted move

The host app saved the reorder or parent move and may keep a short success cue.

treeoutcome cue
movedAccepted child row move saved
  • host success copy
  • final row order

Rejected by policy

The host app rejects a valid transfer because permission, target type, or position rules fail.

treeoutcome cue
rejectCross-project move not allowed
  • authorization
  • validation
  • final copy

Retry needed

The host app could not persist the move and owns the retry, undo, or reload path.

treeoutcome cue
retryUnsaved move retry save
  • network failure
  • host recovery

Transfer boundary states

These static states show how transfer guards and invalid events can be reviewed without making TreeView responsible for final move policy.

  • Use data-tree-transfer-disabled="true" as a representative disabled target cue.
  • Use invalid styling to show the event boundary for malformed row payloads, missing external source payloads, or transferred values.
  • Keep final permission wording, persistence, audit trails, and reorder rules outside this mockup.

Transfer disabled target

The row exposes a disabled transfer guard; authorization and exact explanation stay in the host app.

Invalid row payload
