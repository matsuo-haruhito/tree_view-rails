tree_view-rails

TreeView windowed rendering mock

Static HTML for reviewing representative offset and limit slices, current-row anchoring, and previous or next metadata from windowed rendering. Pagination controls, query state, data fetching, scroll observers, and DOM virtualization remain host-app responsibilities and are intentionally not modeled here.

Mode E

RenderWindow versus virtual scrolling

Use this state when the review goal is to separate TreeView's render metadata from a host-app-owned scroll virtualization layer.

RenderWindow can reduce the number of TreeView rows emitted for a known visible list. It does not observe scroll position, recycle DOM nodes, synchronize URL state, or choose an external virtualization library.

TreeView render control

  • offset, limit, visible row range, and boundary metadata.
  • Current-row anchoring once the host app has chosen the current record.
  • HTML output reduction for rows that are already known to the render pass.

Host app virtualization

  • Scroll observers, viewport measurement, overscan, and DOM recycling.
  • Route or URL state for the current scroll window and restore behavior.
  • External virtualization library setup, fetch timing, and loading placeholders.
1. Host app chooses the scroll policy. It decides whether the screen uses pagination, load-more, an IntersectionObserver, or a virtualization library.
2. Host app maps viewport state to render inputs. When useful, it converts its scroll state into a stable offset and limit for TreeView.
3. TreeView renders the requested window. The output keeps row cues and metadata readable, but final scrolling behavior remains outside the gem contract.
Review question Boundary answer
Can this mock stand in for a virtual scrolling engine? No. It only shows how a render window looks after the host app has chosen which rows to request.
Can a host app combine both approaches? Yes, but the host app owns observer state, overscan, fetch policy, and restore behavior.
Should TreeView add infinite scroll controls here? No. This mock keeps scroll-position-driven controls out of the gem surface.
