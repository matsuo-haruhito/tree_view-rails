Form editing unsaved changes / replacement risk

This mockup makes dirty rows and replacement hazards visible for review without adding TreeView-owned dirty tracking, save flows, Turbo replacement handling, or filtering policy.

Editable rows with risk cues

Host forms own state. TreeView only needs predictable slots and non-overlapping row affordances.

host-owned form state
  • Roadmap Folder row remains stable while child rows are edited saved
  • Sprint planning Dirty row before collapse, filtering, or windowed removal unsaved local edits
  • Customer notes Potential replacement target from lazy loading, Turbo, filter, or virtual window replacement risk

Replacement cases to compare

  1. Collapsed parent hides a dirty descendant.
  2. Lazy-loading replaces the child list while a row has local edits.
  3. Turbo replacement refreshes a frame containing an active form.
  4. Filtering or windowing removes the row from the visible set.

Boundary checklist

form_partial owns inputs, validation copy, and submit/cancel actions.
TreeView can provide stable row slots and spacing for inline controls.
Dirty-state detection, confirmation dialogs, persistence, and recovery remain host app behavior.
The mockup highlights visual risk states only; it is not a runtime state machine.
