tree_view-rails

TreeView children pagination mock

Focused static HTML for reviewing next-page placeholder placement and load-more affordances without running a Rails app. This page keeps cursors, limits, authorization, Turbo Stream responses, and host-app paging logic out of scope on purpose.

Back to review gallery Open mockup index

What this surface is for

  • Compare where the host app places the next-page placeholder row relative to already loaded children.
  • Inspect how a load-more affordance stays visually tied to one parent branch instead of reading like a global toolbar action.
  • Keep the TreeView-owned branch structure visible while leaving cursor semantics and fetch behavior to the host app.

Responsibility boundary reminders

TreeView-owned cues in this mockBranch structure, placeholder-row placement, and the visual distinction between already loaded rows and a branch-scoped next-page affordance.
Host-app decisions kept out of scopeCursor encoding, next-page detection, query limits, authorization, Turbo Stream append or replace behavior, and final button wording.

Review cues

StateBest for checkingKeep outside scope
Initial pageWhether the placeholder row reads as part of the same expanded branch after the first child page loads.Query ordering, cursor rules, or whether page size should be 25, 40, or 50.
Scoped placeholderWhether each branch can own its own load-more affordance without looking like a global toolbar or footer action.Real routes, Turbo Stream targets, authorization-aware button copy, or analytics events.
After appendWhether the placeholder moves downward with the branch as more children arrive, instead of drifting outside the tree context.Actual append logic, DOM IDs, cursor persistence, or retry behavior after failures.
